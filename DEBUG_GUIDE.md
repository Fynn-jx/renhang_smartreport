# 公文写作调试功能使用指南

## 功能说明

公文写作服务现在包含完整的调试功能，可以记录每个阶段的输入输出，帮助你诊断问题并优化提示词。

## 自动生成的文件

每次运行公文写作任务后，会在 `storage/debug_outputs/` 目录下创建一个会话文件夹，包含：

### 📁 会话目录结构

```
storage/debug_outputs/20241203_143022/
├── 00_会话总结.md              # 会话总览（必读）
├── 00_会话总结.json            # 机器可读的JSON格式
├── 20241203_143022_001_01_大纲生成.md
├── 20241203_143045_002_02_章节写作_一、现行财政框架的局限性.md
├── 20241203_143048_003_02_章节写作_二、国际财政规则实践经验.md
├── 20241203_143102_004_05_内容整合.md
├── 20241203_143120_005_06_最终调整.md
└── ...
```

### 📊 会话总结 (00_会话总结.md)

**这是最重要的文件**，包含：

1. **阶段概览表格**：每个阶段的模型、输入输出长度
2. **统计信息**：总输入、总输出、压缩率
3. **潜在问题检测**：自动标记异常的阶段
   - ⚠️ 输出异常膨胀（超过输入1.5倍）
   - ⚠️ 输出异常收缩（小于输入30%）
   - ❌ 输出过短（少于100字符）
4. **快速链接**：点击跳转到每个阶段的详细文件

### 📝 各阶段详细文件

每个阶段的详细文件���含：

- **输入**：完整的输入内容（最多8000字符）
- **提示词**：使用的提示词（最多3000字符）
- **输出**：完整的输出内容（最多10000字符）
- **元数据**：模型名称、长度统计、压缩率等

## 使用方法

### 1. 启用调试模式

调试模式默认已启用，无需额外配置。

### 2. 运行公文写作任务

正常使用前端或API运行公文写作转换。

### 3. 查看调试输出

#### 方法1：直接查看文件

```bash
# 查看最新会话目录
cd storage/debug_outputs
ls -lt | head -5

# 进入最新会话目录
cd 20241203_143022/

# 查看会话总结
cat 00_会话总结.md

# 或使用VSCode打开
code .
```

#### 方法2：在浏览器中查看

由于文件是Markdown格式，可以直接在浏览器或VSCode中打开查看。

### 4. 分析问题

#### 查看会话总结

首先打开 `00_会话总结.md`，查看：

1. **阶段概览表格** - 快速了解每个阶段的情况
2. **潜在问题检测** - 查看是否有自动检测到的问题
3. **统计信息** - 了解整体压缩率

#### 对比不同阶段

点击表格中的文件链接，查看具体阶段的输入输出。

**常见问题诊断**：

| 问题 | 查看哪个文件 | 寻找什么 |
|------|------------|---------|
| 大纲结构不合理 | `01_大纲生成.md` | 输出的框架是否清晰 |
| 章节内容空洞 | `02_章节写作_*.md` | 输出长度是否过短 |
| 内容整合后丢失 | `05_内容整合.md` | 对比输入输出 |
| 最终格式混乱 | `06_最终调整.md` | 检查格式变化 |

## 实际案例分析

### 案例1：内容整合后文字大量丢失

**现象**：
- 输入：45,000 字符
- 输出：7,000 字符
- 压缩率：84%

**诊断步骤**：

1. 打开 `05_内容整合.md`
2. 查看"输入"部分 - 确认原始内容完整
3. 查看"输出"部分 - 检查是否真的丢失了内容
4. 查看提示词部分 - 检查是否有"精简"相关指令

**解决方案**：
- 检查 `INTEGRATION_SYSTEM_PROMPT` 是否过度强调精简
- 调整提示词，明确"完整保留实质性内容"

### 案例2：章节写作内容空洞

**现象**：
- 某个章节输出只有200字符
- 同样长度的其他章节有1000字符

**诊断步骤**：

1. 打开对应章节的 `02_章节写作_*.md`
2. 查看"输入"部分 - 确认输入信息足够
3. 查看"输出"部分 - 检查是否敷衍了事
4. 对比其他章节的输入输出

**解决方案**：
- 检查章节标题是否过于抽象
- 增强 `CHAPTER_WRITING_PROMPT` 的具体性要求
- 在输入中添加更多上下文

### 案例3：大纲生成不符合公文规范

**现象**：
- 大纲中出现了"一、二、三、"
- 但没有"摘要/按"部分

**诊断步骤**：

1. 打开 `01_大纲生成.md`
2. 查看"输出"部分 - 检查结构
3. 查看"提示词"部分 - 确认是否明确要求了"摘要/按"

**解决方案**：
- 修改 `OUTLINE_SYSTEM_PROMPT`
- 强调必须包含"摘要/按"部分
- 提供公文结构示例

## 优化提示词的流程

1. **运行一次公文写作任务**
2. **查看会话总结**，了解整体情况
3. **逐个阶段检查**，找到问题阶段
4. **分析输入输出**，理解问题原因
5. **修改对应的提示词**
6. **重新运行**，验证修改效果
7. **对比两次的调试输出**，确认改善

## 高级用法

### 对比两次运行

```bash
# 运行1
cd storage/debug_outputs/20241203_143022/
cat 00_会话总结.md

# 运行2（修改提示词后）
cd storage/debug_outputs/20241203_150833/
cat 00_会话总结.md

# 对比
diff ../20241203_143022/00_会话总结.md 00_会话总结.md
```

### 统计分析

使用JSON文件进行自动化分析：

```python
import json

# 读取两次运行的统计
with open('20241203_143022/00_会话总结.json', 'r', encoding='utf-8') as f:
    run1 = json.load(f)

with open('20241203_150833/00_会话总结.json', 'r', encoding='utf-8') as f:
    run2 = json.load(f)

# 对比
print(f"运行1压缩率: {run1['statistics']['compression_rate']:.1f}%")
print(f"运行2压缩率: {run2['statistics']['compression_rate']:.1f}%")
```

## 清理调试文件

调试文件会占用磁盘空间，建议定期清理：

```bash
# 删除7天前的调试记录
find storage/debug_outputs -type d -mtime +7 -exec rm -rf {} \;

# 或保留最近10次
cd storage/debug_outputs
ls -t | tail -n +11 | xargs rm -rf
```

## 禁用调试模式

如果不需要调试功能，可以在代码中禁用：

```python

debug_logger = WorkflowDebugLogger(enabled=False)  # 改为 False
```

## 故障排除

### 问题：没有生成调试文件

**原因**：
- 调试模式被禁用
- 文件权限问题
- 磁盘空间不足

**解决**：
- 检查 `debug_mode` 是否为 `True`
- 检查 `storage/debug_outputs/` 目录权限
- 检查磁盘空间

### 问题：文件内容不完整

**原因**：
- 内容过长被截断（保护机制）

**解决**：
- 查看实际完整文件（不在Markdown中，而在原始文件中）
- 或增加 `_write_content` 方法中的 `max_chars` 参数

## 技术细节

### 文件命名格式

```
{时间戳}_{序号}_{阶段名称}.md

例如：20241203_143022_001_01_大纲生成.md
├─ 20241203_143022: 完整时间戳
├─ 001: 序号（3位数字，不足补0）
└─ 01_大纲生成: 阶段名称
```

### 内容截断机制

为了防止文件过大，某些内容会被截断：

- 输入内容：最多8000字符
- 提示词：最多3000字符
- 输出内容：最多10000字符

实际完整内容保存在原始变量中，只是显示时截断。
